## Phase 2 — Tab-by-tab financial-logic review (COMPLETE)

Delegated the deep-read to an Explore agent. Then human-verified the one CRITICAL claim the agent raised. Result: one false positive (instructive), three MINOR findings.

### False-positive finding (instructive, NOT a bug)

**Agent claim.** `Expense Summary!EXP_NACQ` omits `UWEX_GFFE`; since `UW Exec Summary!UWEX_NACQ` includes `UWEX_GFFE`, the Income Statement understates expenses → Net Income overstated.

**Actual.** Verified by algebra. Fronting fees flow to Income Statement via **Revenue Summary**, not Expense Summary:
- `REV_FFE = -UWEX_GFFE` → positive value on revenue side
- `REV_UWREV = REV_NEP + REV_FFE` → includes fronting fee as fee income
- `EXP_NACQ = UWEX_GCOMM + UWEX_CEDCOMM` → net commission only, no fronting

Operating Income from UW portion:
```
(NEP + FFE) − (NCOMM + NLLAE + RIEXP)
= NEP − NLLAE − NCOMM − RIEXP + FFE
```
UWEX_UWINC:
```
NEP − NLLAE − NACQ − RIEXP
= NEP − NLLAE − (NCOMM + GFFE) − RIEXP   [where GFFE is stored negative]
= NEP − NLLAE − NCOMM + FFE − RIEXP
```
Identical. Net Income reconciles.

**Why the agent got it wrong.** Pattern-matching: `UWEX_NACQ` and `EXP_NACQ` have different structures. A naive audit assumes parallel structure = parallel math. Here, the two perspectives are equivalent through a legitimate accounting choice:
- UW Exec view: net fronting fee into acquisition cost (reduces net acquisition → increases UW income).
- Income Statement view: separate fronting fee into revenue line; expense shows gross commission only.

**Recommendation.** Add a comment to `EXP_NACQ` formula in `formula_tab_config.csv` explaining the split, so future auditors (human or agent) don't re-flag it. Low effort. Filed below as MINOR-1.

### MINOR findings (no immediate action; documented for future maintenance)

**MINOR-1 — Parallel-structure divergence between `UWEX_NACQ` and `EXP_NACQ`.**
Category: Auditability (#5).
Location: `config/formula_tab_config.csv:1200` vs `:1404`.
Issue: `UWEX_NACQ` includes fronting fee; `EXP_NACQ` does not. The accounting is correct (fronting flows via revenue), but a naive reader would expect parallel structure.
Suggested fix: Add comment field to both cells explaining the split. Optionally add a linked comment to `REV_FFE` saying "fronting fee counted as revenue here; net-of-fronting acquisition cost appears in Expense Summary row 6."
Effort: 5 min.

**MINOR-2 — Investment pool circular dependency.**
Category: Robustness (#4).
Location: `config/formula_tab_config.csv` — `INV_Q_POOL` formula (Investments tab).
Issue: `INV_Q_POOL = BS_TOTAL_LE − other asset lines`. `BS_TOTAL_LE` includes current-period `BS_EQUITY`, which includes current-period `IS_NETINC`, which includes current-period `INV_Q_INC_NET` — which is derived from `INV_Q_POOL`. Cycle.
Mitigation in place: investment income formulas use `PREV_Q:INV_Q_INVESTED` (prior quarter's invested balance) for yield calc. This breaks the cycle in principle — each quarter's investment income depends on prior quarter's pool, not current. But Excel may still detect a circular reference if iterative calculation isn't enabled.
Suggested fix: Verify `Application.Iteration = True` is set at bootstrap OR refactor `INV_Q_POOL` to compute from prior-quarter values only. Document either choice explicitly.
Effort: 30 min investigation + 15 min fix.

**MINOR-3 — Undercapitalization flag absent.**
Category: Robustness (#4).
Location: Investments tab, `INV_Q_POOL` formula.
Issue: If `BS_EQUITY` goes negative (catastrophic scenario), `INV_Q_POOL` could go negative. Comment in the code hints at this but no visible flag or handler.
Suggested fix: Add a validation row `INV_Q_CHECK = IF(INV_Q_POOL < 0, "NEGATIVE POOL", "OK")` visible on the tab. Alternatively, surface via the Health/Diagnostic tab if one exists.
Effort: 15 min.
