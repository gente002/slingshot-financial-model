# Design Strategy: Kernel vs Domain

## Core Principle

**The kernel is a compiler. Config is the source code. The domain module is a plugin.**

| Layer | Role | Who writes it | Examples |
|-------|------|---------------|----------|
| Kernel | Knows HOW | Framework authors | KernelEngine, KernelFormulaWriter, KernelBootstrap |
| Config | Tells kernel WHAT | Model developer | formula_tab_config, tab_registry, column_registry |
| Domain | Implements COMPUTATION | Model developer | InsuranceDomainEngine, MonteCarloDomainEngine |
| UI | The OUTPUT | Generated by kernel | Excel workbook (tabs, formulas, charts) |

## What Goes in the Kernel

Kernel modules are **generic infrastructure**. They work for ANY model — insurance, Monte Carlo, ETL, SaaS. A kernel module should NEVER contain:

- Company names or branding (use branding_config.csv)
- Domain-specific tab names (read from tab_registry)
- Domain-specific RowIDs or column names (read from formula_tab_config, column_registry)
- Domain-specific validation rules (use validation_config.csv)
- Domain-specific text or labels (use msgbox_config.csv, display_aliases.csv)
- Hardcoded references to domain modules (use DomainModule setting, extension_registry)

**Test:** If you replaced all config CSVs with a Monte Carlo model's config, would the kernel still compile and run? If any kernel module would break, it has a domain leak.

## What Goes in Config

Config CSVs define the model. Adding a metric, a tab, a formula, a validation rule, or a health check = adding a row to a CSV. No VBA editing required.

| Config File | What It Defines |
|-------------|----------------|
| tab_registry.csv | Which tabs exist, their order, visibility, type |
| formula_tab_config.csv | Every cell on every formula-driven tab |
| column_registry.csv | Detail tab columns (dimensions, metrics, derived fields) |
| input_schema.csv | Global assumption parameters |
| validation_config.csv | Input validation rules per tab/RowID |
| health_config.csv | Balance check conditional formatting |
| pipeline_config.csv | Which pipeline steps are enabled |
| branding_config.csv | Company name, model title, post-run navigation |
| named_range_registry.csv | Excel named ranges for cross-tab references |

## What Goes in the Domain Module

The domain module implements the **computation contract** — the math that turns inputs into outputs. For insurance: actuarial development curves, premium earning patterns, loss reserving. For Monte Carlo: random sampling, variance reduction, convergence testing.

**Domain contract (required functions):**
- `Initialize()` — read inputs, set up state
- `Validate() As Boolean` — pre-run checks
- `Reset()` — clear state
- `Execute()` — fill DomainOutputs array

**Domain companion modules** (optional): Ins_QuarterlyAgg, Ins_Triangles, Ins_Tests, Ins_Presentation. These implement domain-specific features that the kernel dispatches via config (extension_registry, branding_config).

## Design Patterns

### Pattern 1: Domain Defines, Kernel Processes

The domain author writes config. The kernel reads config and executes.

Example: ProveIt audit checks
- Domain defines checks in `prove_it_config.csv` (CheckID, MetricA, MetricB, Operator)
- Kernel (`KernelProveIt.bas`) reads config and generates Excel verification formulas

This pattern applies to: formula tabs, validation rules, health checks, exhibits, charts, pipeline steps, branding, named ranges.

### Pattern 2: Committed State via Run Metadata

Input tabs are staging areas — the user may be editing them for the next run. Downstream formula tabs must NOT reference input tabs directly. Instead:

1. User enters inputs on input tabs (UW Inputs, Capital Activity, etc.)
2. User clicks Run Model
3. Kernel commits input state to the **Run Metadata** tab (hidden)
4. Downstream tabs reference Run Metadata via named ranges (ENTITY_1..10)

This prevents mid-edit input changes from corrupting displayed results.

### Pattern 3: Config-Driven Dispatch

The kernel never hardcodes which domain module to call. Instead:

- `granularity_config.csv` defines `DomainModule=InsuranceDomainEngine`
- `branding_config.csv` defines `DomainTestEntry=Ins_Tests.RunInsuranceTests`
- `extension_registry.csv` defines PostBootstrap, PostOutput, DashboardButton hooks

The kernel uses `Application.Run(domMod & ".FunctionName")` for the domain contract, and `KernelExtension.RunExtensions` for hooks.

### Pattern 4: Tab Existence is Config-Driven

The kernel only creates tabs listed in `tab_registry.csv`. It hardcodes exactly 4 infrastructure tabs (Config, Dashboard, Detail, ErrorLog). All others — including Summary, ProveIt, Exhibits, Charts — are optional. If a model doesn't need them, omit them from tab_registry.

Kernel modules that operate on optional tabs check for existence first:
```vba
Set ws = Nothing
On Error Resume Next
Set ws = ThisWorkbook.Sheets(tabName)
On Error GoTo 0
If ws Is Nothing Then Exit Sub  ' tab not in this model
```

### Pattern 5: No Hardcoded Cell References Between Tabs

Formula tabs must NOT use `='Tab Name'!C5` style references. Instead:
- Use `{NAMED:RangeName}` for named range references
- Use `{REF:Tab!RowID}` for cross-tab RowID references
- Use `{ROWID:RowID}` for same-tab RowID references

Named ranges are defined in `named_range_registry.csv` and point to specific tabs/cells. If a tab is renamed or reorganized, update the named range — formula_tab_config doesn't change.

## Anti-Patterns

1. **Kernel code references a domain tab name** — use tab_registry or branding_config
2. **Kernel code references a domain RowID** — use health_config or validation_config
3. **Formula references input tab directly** — use Run Metadata + named ranges
4. **Domain logic in kernel module** — move to domain module, dispatch via config
5. **Hardcoded EnsureSheet for optional tabs** — let tab_registry be the authority
6. **Per-entity columns for global parameters** — global params get one cell, not one per entity
