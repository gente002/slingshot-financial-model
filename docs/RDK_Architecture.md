# RDK Architecture

## Four-Layer Stack

```
+--------------------------------------------------+
|                    UI LAYER                       |
|  Excel Workbook (tabs, formulas, charts, buttons) |
|  Generated from Configuration by kernel           |
+--------------------------------------------------+
|              DOMAIN LOGIC LAYER                   |
|  DomainEngine.bas (one module per model)          |
|  Implements: Initialize, Validate, Reset, Execute |
|  Companion modules: Ins_*.bas (optional helpers)  |
+--------------------------------------------------+
|            CONFIGURATION LAYER                    |
|  CSV files (config_<model>/ directory)            |
|  Tells kernel WHAT to build                       |
|  Adding a metric = one config row                 |
+--------------------------------------------------+
|               KERNEL LAYER                        |
|  KernelXxx.bas modules (24 modules)               |
|  Knows HOW to build, never edited by developer   |
|  Generic infrastructure for any financial model   |
+--------------------------------------------------+
```

## Core Principle

**The kernel is a compiler. Config is the source code. The domain module is a plugin.**

- Kernel knows HOW (tab creation, formula generation, CSV I/O, error logging)
- Config tells kernel WHAT (which tabs, which formulas, which validations)
- Domain module implements the COMPUTATION (actuarial math, Monte Carlo, ETL)
- UI is the OUTPUT (Excel workbook generated by kernel from config)

## Domain Contract

Every domain module must implement these well-known functions:

| Function | Signature | Required | Purpose |
|----------|-----------|----------|---------|
| Initialize | () | Yes | Set up domain state from input parameters |
| Validate | () As Boolean | Yes | Pre-run checks; return False to halt pipeline |
| Reset | () | Yes | Clear computation state before Execute |
| Execute | () | Yes | Fill outputs via DomainOutputs (Incremental + Dimension cols) |
| GetRowCount | () As Long | Optional | Total Detail rows (default: entities x periods) |
| GetMaxPeriod | () As Long | Optional | Max period for Summary columns |
| GetEntityCount | () As Long | Optional | Entity count for metadata |
| RunDomainTests | () As Boolean | Optional | Domain validation tests |
| GranularCSVPath | () As String | Optional | Path to granular export file |

The kernel discovers the domain module name from `DomainModule` setting in
`granularity_config.csv` and dispatches via `Application.Run(domMod & ".FunctionName")`.

## Pipeline

The pipeline is config-driven via `pipeline_config.csv`. Each step has:
- StepID: unique identifier
- StepOrder: execution order (lower = first)
- Enabled: TRUE/FALSE (skip disabled steps)

The kernel engine reads pipeline_config and executes enabled steps in order.
A Monte Carlo model can disable WRITE_CSV and WRITE_SUMMARY if not needed.

```
LOAD_CONFIG --> DOMAIN_INIT --> VALIDATE --> COMPUTE --> WRITE_DETAIL
    --> WRITE_CSV --> WRITE_SUMMARY --> REFRESH_* --> DOMAIN_TESTS --> METADATA
```

## Config Tables (24)

### Kernel Infrastructure (6)
- column_registry.csv: Detail tab column definitions
- input_schema.csv: Input parameter definitions
- granularity_config.csv: Time axis, DomainModule, entity settings
- tab_registry.csv: Tab definitions (name, type, visibility, sort order)
- repro_config.csv: PRNG seed, deterministic mode
- scale_limits.csv: Row/column limits, CSV-only thresholds

### Model Definition (6)
- formula_tab_config.csv: Formula tab layouts with placeholder resolution
- named_range_registry.csv: Named range definitions
- summary_config.csv: Summary tab metrics
- prove_it_config.csv: Audit check definitions
- pipeline_config.csv: Pipeline step sequence and enable/disable

### Presentation (5)
- chart_registry.csv: Dashboard chart definitions
- exhibit_config.csv: Audit exhibit definitions
- display_mode_config.csv: Incremental/Cumulative toggle
- print_config.csv: Print/PDF layout settings
- branding_config.csv: Company name, model title, dashboard metrics

### Extensions (2)
- extension_registry.csv: Extension module registration and hooks
- curve_library_config.csv: Curve parameters (CurveLib extension)

### Quality (3)
- validation_config.csv: Input validation rules
- health_config.csv: Health check conditional formatting rules
- config_schema.csv: Machine-readable schema for all config tables

### Infrastructure (2)
- data_model_config.csv: Power Pivot settings
- pivot_config.csv: Pivot table definitions

## Extension System

Extensions are optional add-on modules that hook into the pipeline at defined points:

| HookType | When | Example |
|----------|------|---------|
| PreCompute | Before domain Execute | Data preprocessing |
| PostCompute | After domain Execute | Result validation |
| PostOutput | After Detail/Summary written | Report generation |
| PostBootstrap | After bootstrap completes | Reference tab building |
| DashboardButton | Bootstrap Dashboard setup | UI buttons for extensions |
| Standalone | Not pipeline-triggered | Utility libraries |

Extensions register in `extension_registry.csv` and are dispatched by `KernelExtension.RunExtensions`.

## Snapshot System

Snapshots capture scenario state for save/restore:
- config/*.csv: Full model configuration
- inputs.csv: User input values from Assumptions tab
- detail.csv: Computation results
- settings.csv: Run state metadata
- manifest.json: Version, hashes, fingerprint

Load All: restores config + inputs + results (instant, frozen results).
Load Inputs: restores config + inputs only (must re-run for fresh results).

## File Organization

```
  config/            Runtime config (seeded from config_<model>/)
  config_insurance/  Insurance FM model config

  config_blank/      Blank bootable config skeleton

  docs/              Design specs, roadmaps, architecture
  snapshots/         Saved scenario state
  output/            Exported files (granular CSV, reports)
  scripts/           Setup and utility scripts
```
